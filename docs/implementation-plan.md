# План реализации и защиты

## Цель на максимум баллов

Показать не “чат к таблице”, а управляемую аналитическую платформу с воспроизводимым NL→SQL pipeline, безопасностью, explainability и аккуратным UX.

## Этапы реализации

1. Поднять PostgreSQL и загрузить demo-датасет.
2. Проверить semantic layer и типовые русскоязычные сценарии.
3. Запустить `llm` сервис в режиме `rule-based` и отладить parse flow.
4. Подключить `GigaChat` или `YandexGPT` и переключить `LLM_PROVIDER`.
5. Отладить guardrails в `query` сервисе.
6. Проверить сохранение отчётов и повторный запуск.
7. Подготовить четыре безошибочных демосценария и один ambiguous-сценарий.

## Что показывать на защите

- Один понятный вопрос менеджера.
- Экран интерпретации намерения.
- SQL preview и объяснение, почему он безопасный.
- График и таблицу с результатом.
- Сохранение отчёта и повторный запуск.
- Короткий рассказ о том, как решение масштабируется после MVP.

## Вопросы, к которым стоит быть готовым

- Почему LLM не генерирует SQL напрямую?
- Как защищаетесь от опасных запросов?
- Что делать с неоднозначными формулировками?
- Как подключить реальную схему организаторов после полного раскрытия кейса?
- Как масштабировать сервис при росте нагрузки?

## Ответы для экспертов

- SQL строится только backend-ом из структурированного `intent JSON`, поэтому контроль остаётся у системы.
- Guardrails встроены на уровне структуры intent, allowlist колонок, проверки SQL и read-only роли БД.
- Неоднозначность закрывается через `clarification` и `confidence score`.
- Для реальной схемы достаточно заменить источник `analytics.v_ride_metrics` на представления или materialized views организаторов.
- При росте нагрузки можно масштабировать `query` и `llm` отдельно, добавить кэширование, очередь на тяжёлые запросы и precomputed datasets.
