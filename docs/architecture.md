# Архитектура Drivee Analytics MVP

## Цель

Собрать self-service аналитическую платформу, в которой бизнес-пользователь задаёт вопрос по-русски и получает воспроизводимый аналитический результат без знания SQL.

## Контур решения

1. `Gateway` отдаёт frontend и проксирует запросы во внутренние сервисы.
2. `Meta Service` публикует разрешённые метрики, измерения, словарь терминов и шаблоны вопросов.
3. `LLM Service` переводит русский текст в структурированный `intent JSON`.
4. `Query Service` валидирует intent, строит SQL по whitelist-правилам, исполняет запрос и возвращает explainability.
5. `Reports Service` сохраняет отчёты, историю запусков и переиспользует готовые сценарии.
6. `PostgreSQL` хранит аналитический факт и служебные таблицы приложения.

## Почему микросервисный подход здесь уместен

- Хорошо ложится на критерий “реализуемость MVP” и показывает зрелое проектирование.
- Удобно масштабировать отдельно тяжёлый контур `query`, нестабильный внешний контур `llm` и пользовательский контур `gateway`.
- Позволяет независимо развивать semantic layer, отчёты и future-функции вроде рассылок.

## Frontend

- Светлый продуктовый интерфейс в стиле drivee: крупные карточки, lime-акценты, округлые блоки, минимум визуального шума.
- Главный экран объединяет ввод запроса, explainability и результат без лишней навигации.
- Пользователь видит:
  - интерпретацию вопроса;
  - confidence score;
  - безопасный SQL preview;
  - таблицу и график;
  - сохранённые отчёты.

## Backend pipeline

1. Пользователь отправляет текстовый запрос в `Gateway`.
2. `Query Service` запрашивает актуальный semantic layer у `Meta Service`.
3. `Query Service` отправляет текст и semantic layer в `LLM Service`.
4. `LLM Service` вызывает `GigaChat` или `YandexGPT`; если ключей нет, работает локальный `rule-based` parser.
5. На выходе всегда получается структурированный `intent JSON`, а не готовый SQL.
6. `Query Service` строит SQL сам, применяя allowlist-правила по метрикам, измерениям и фильтрам.
7. Финальный SQL проходит security-check и выполняется в PostgreSQL.
8. Результат логируется, обогащается chart-метаданными и возвращается во frontend.

## База данных

### Аналитический слой

- `analytics.ride_metrics_daily` — денормализованный факт для MVP-демо.
- `analytics.v_ride_metrics` — представление, через которое работает Query Service.

### Служебный слой

- `app.saved_reports` — сохранённые пользовательские отчёты.
- `app.report_runs` — история повторных запусков.
- `app.query_logs` — журнал выполнения и ошибок.
- `app.semantic_terms` — словарь терминов для бизнес-сленга и explainability.

## Guardrails

- LLM никогда не получает прямой доступ к БД.
- SQL строится только сервером Go.
- Поддерживается только `SELECT`.
- Фильтры и группировки проходят через allowlist.
- В БД предусмотрена отдельная read-only роль `analytics_readonly`.
- Все запросы и ошибки журналируются в `app.query_logs`.

## Работа с русской LLM

- Основной провайдер: `GigaChat`.
- Альтернатива: `YandexGPT` как российский стек с качественной работой по русскому языку.
- Архитектурное правило: модель отвечает только за понимание запроса и формирование структуры намерения.
- Даже при использовании модели финальное решение остаётся детерминированным благодаря builder-логике на Go.
